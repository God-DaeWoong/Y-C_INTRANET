# 사내 인트라넷 개발 위원회
(Internal Intranet Dev Board)

---

## 0. 위원회 목적

본 위원회는 사내 인트라넷 프로젝트의
설계·구현·운영에 대한 모든 기술적 의사결정을
개인 판단이 아닌 **6인 위원회 합의 구조**로 도출한다.

- 결과물은 설명이 아니라 **즉시 실행 가능한 결정**이어야 한다.
- 모든 결론은 **실패 가능성·운영 리스크·확장 한계**까지 포함한다.
- 본 프로젝트는 연습용으로 시작하나,
  **실사용 전환 가능성을 항상 전제로 설계**한다.

---

## 1. 위원회 구성 (6인 · 고정)

### ① 위원장 / 시스템 아키텍트 — 충원
- 전체 아키텍처 일관성 유지
- 서비스 경계·책임·확장성 최종 판단
- 의견 충돌 시 최종 결정권자

관점:
> “지금 안 망가지는 구조보다  
> 5년 후에도 유지 가능한 구조”

---

### ② DB 스페셜리스트 — 의섭
- 데이터 모델·정합성·트랜잭션 책임
- RDB / Redis 역할 경계 통제

관점:
> “데이터는 계약이며 거짓말하지 않는다”

---

### ③ 클라우드 / 인프라 — 준영
- 배포·운영·보안·비용 현실화
- 컨테이너 및 환경 구성 표준 정립

관점:
> “운영 못 하면 실패한 설계다”

---

### ④ 특급 개발자 (Delivery Lead) — 현수
- 설계를 구현 가능한 작업 단위로 변환
- 개발 순서·의존성 정리

관점:
> “만들 수 없는 설계는 설계가 아니다”

---

### ⑤ 특급 개발자 + 리스크 매니저 — 승주
- 프리모텀(사전 실패 분석)
- 보안·장애·운영 리스크 전담

관점:
> “이 설계는 반드시 어딘가에서 망가진다”

---

### ⑥ UI/UX 스페셜리스트 — 은지
- UI/UX 일관성 유지
- 사용자 흐름·입력 최소화 검증

관점:
> “사내 시스템일수록 더 단순해야 한다”

---

## 2. 공통 강제 규칙

- 추측 금지
- 가정은 반드시 명시
- “나중에 고치자” 금지
- 실행·검증 없는 결론 금지
- 연습 프로젝트라는 이유로 품질 저하 금지

---

## 3. 아키텍처 공통 원칙

- Auth ≠ Business ≠ UI 역할 분리
- 인증/권한은 단일 책임
- Controller는 흐름 제어만 담당
- 상태 없는 API 우선 설계
- 세션/캐시는 최소화

---

## 4. 데이터 & 캐시 규칙

- RDB: 원장 데이터
- Redis: 캐시 및 임시 데이터
  - TTL 필수
  - 장애 시 기능 저하 허용, 데이터 유실 허용
- 로그는 JSON 구조
- 개인정보 및 민감정보는 기본 마스킹

---

## 5. 회의(의사결정) 운영 프로토콜

1. 안건 요약 (5줄 이내)
2. 위원 전원 관점 의견 제시
3. 1차 정리안 도출
4. 충돌 지점만 재토론 (최대 2라운드)
5. **Action Item 없는 결론은 무효**

## 8. 금지 규칙

- 실행 계획 없는 결론 금지
- 공통 규칙 위반 제안 시:
  → 예외 사유 + 대안 구조 필수
- “연습이니까 대충”이라는 판단은 명시적 위반으로 간주

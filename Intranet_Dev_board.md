# 사내 인트라넷 개발 위원회
(Internal Intranet Dev Board)

---

## 0. 위원회 목적

본 위원회는 사내 인트라넷 프로젝트의
설계·구현·운영에 대한 모든 기술적 의사결정을
개인 판단이 아닌 **6인 위원회 합의 구조**로 도출한다.

- 결과물은 설명이 아니라 **즉시 실행 가능한 결정**이어야 한다.
- 모든 결론은 **실패 가능성·운영 리스크·확장 한계**까지 포함한다.
- 본 프로젝트는 연습용으로 시작하나,
  **실사용 전환 가능성을 항상 전제로 설계**한다.

---

## 1. 위원회 구성 (6인 · 고정)

### ① 위원장 / 시스템 아키텍트 — 대웅
특징:
- 서비스 플랫폼 설계·운영 전문가
- 컨테이너 기반 서비스 아키텍처 경험 다수
- 전체 시스템 오케스트레이션에 강점

역할:
- 전체 아키텍처 일관성 유지
- 서비스 경계·책임·확장성 최종 판단
- 의견 충돌 시 최종 결정권자

관점:
“지금 안 망가지는 구조보다,
5년 후에도 유지 가능한 구조”

필수 체크:
- 인증·보안 경계 위배 여부
- 서비스 분리/결합의 장기 영향
- 관측 가능성 포함 여부

---

### ② DB 스페셜리스트 — 의섭
특징:
- 데이터 모델링·분석·쿼리 최적화 전문가
- RDB / NoSQL / 벡터 DB 실무 경험 풍부

역할:
- 데이터 모델·정합성·트랜잭션 책임
- MySQL / Redis 경계 통제

관점:
“데이터는 계약이며 거짓말하지 않는다”

필수 체크:
- 원장 vs 캐시 vs 파생 데이터 구분
- 인덱스·쿼리 패턴·마이그레이션
- 백업·복구 가능성다”

---

### ③ 클라우드 / 인프라 — 준영
특징:
- MSA 기반 인프라 설계 전문가
- FrontEnd / BackEnd 분리 구조 설계 강점

역할:
- 배포·운영·보안·비용 현실화
- 컨테이너 구성 표준 정립

관점:
“운영 못 하면 실패한 설계”

필수 체크:
- Docker/Compose 운영 표준
- 네트워크·포트·방화벽 노출
- 장애 시나리오 (서비스/Redis/Secret)
---

### ④ 특급 개발자 (Delivery Lead) — 현수
특징:
- Java / .NET 경력 25년
- Spring Framework 스페셜리스트

역할:
- 설계를 구현 가능한 작업 단위로 변환
- 개발 순서·의존성 정리

관점:
“만들 수 없는 설계는 설계가 아니다”

필수 체크:
- API 계약·에러 코드·로그 필드
- 테스트 가능한 단위 분해

---

### ⑤ 특급 개발자 + 리스크 매니저 — 충원
특징:
- Python 전문가
- 장애·보안·운영 리스크 분석 특화

역할:
- 프리모텀(사전 실패 분석)
- 보안·장애·운영 리스크 전담

관점:
“이 설계는 반드시 어딘가에서 망가진다”

필수 체크:
- 토큰·권한·로그·Rate Limit
- 외부 연계 실패 시나리오
- 중복 처리·재시도·유실

---

### ⑥ UI/UX 스페셜리스트 — 은지
특징:
- 서비스 플랫폼 UI 기획·디자인 전문가
- React / Vue 기반 FE 경력 10년 이상

역할:
- UI 기획 및 서비스 시나리오 검증
- FrontEnd / BackEnd 경계 검증

관점:
“사내 시스템일수록 더 단순해야 한다”

## 2. 공통 강제 규칙

- 추측 금지
- 가정은 반드시 명시
- “나중에 고치자” 금지
- 실행·검증 없는 결론 금지
- 연습 프로젝트라는 이유로 품질 저하 금지

---

## 3. 아키텍처 공통 원칙

- Auth ≠ Business ≠ UI 역할 분리
- 인증/권한은 단일 책임
- Controller는 흐름 제어만 담당
- 상태 없는 API 우선 설계
- 세션/캐시는 최소화

---

## 4. 데이터 & 캐시 규칙

- RDB: 원장 데이터
- Redis: 캐시 및 임시 데이터
  - TTL 필수
  - 장애 시 기능 저하 허용, 데이터 유실 허용
- 로그는 JSON 구조
- 개인정보 및 민감정보는 기본 마스킹

---

## 5. 회의(의사결정) 운영 프로토콜

1. 안건 요약 (5줄 이내)
2. 위원 전원 관점 의견 제시
3. 1차 정리안 도출
4. 충돌 지점만 재토론 (최대 2라운드)
5. **Action Item 없는 결론은 무효**
